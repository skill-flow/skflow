# Design: sh-error-recovery

## Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│  executeLoop (runner.ts)                                        │
│                                                                 │
│  while (true) {                                                 │
│    result = step(state, input)        ← may throw               │
│                                                                 │
│    if (isShYield(result)) {                                     │
│      shResult = execSh(result._sh)                              │
│      try {                                                      │
│        nextResult = step(result.next, JSON.stringify(shResult))  │
│      } catch (err) {                                            │
│        if (isShError(err)) {                                    │
│          ─── NEW: yield sh-error ───                            │
│          saveState(sessionId, result.next)  // stay at same phase│
│          return { yield: { type: "sh-error", ... } }            │
│        }                                                        │
│        throw err                                                │
│      }                                                          │
│    }                                                            │
│  }                                                              │
│                                                                 │
│  on resume with sh-error session:                               │
│    if (answer === "__abort__") → remove session, return error   │
│    else → execSh(answer) → step(savedState, newResult)          │
└─────────────────────────────────────────────────────────────────┘
```

## Detection: isShError

The `sh-throws` mechanism throws the `ShResult` object directly (augmented with `.cmd`):

```javascript
// Generated by transform:
if (state.x.code !== 0) {
  state.x.cmd = "git push";
  throw state.x;
}
```

Detection at runtime:

```typescript
function isShError(err: unknown): err is ShResult & { cmd: string } {
  return (
    err !== null &&
    typeof err === "object" &&
    "code" in err &&
    "stderr" in err &&
    "cmd" in err &&
    typeof (err as any).code === "number"
  );
}
```

This distinguishes sh errors from generic JS errors (TypeError, ReferenceError, etc).

## State preservation for retry

When yielding `sh-error`, the session state must remain at the "resume after sh()" phase (N+1). This is the same state that was already saved by `saveState(sessionId, result.next)` in the sh auto-resume path. No extra save needed — the state is already correct.

On resume:

1. Load state (phase N+1)
2. Execute new command: `execSh(answer)`
3. Feed result to step: `step(loadedState, JSON.stringify(newShResult))`
4. If step throws again (new command also failed) → yield sh-error again
5. If step succeeds → continue normal execution loop

## Session metadata extension

Add a flag to indicate the session is paused on an error (so resume knows to treat the answer as a command):

```typescript
// meta.json
{
  "scriptName": "deploy",
  "scriptPath": "...",
  "createdAt": 1234567890,
  "pendingShError": {           // NEW — present only during sh-error yield
    "cmd": "git push origin main",
    "result": { "stdout": "", "stderr": "...", "code": 1 }
  }
}
```

When `pendingShError` is present in meta, the resume handler knows:

- answer `"__abort__"` → terminate
- any other answer → treat as new shell command to execute

## Source map generation

### In transform (emit.ts)

Track phase→line mappings during `explodeBody`:

```typescript
interface PhaseMapEntry {
  phase: number;
  line: number;
  source: string | null; // original source line text
  type: string; // "sh", "sh-resume", "ask", "ask-resume", "done"
}
```

Collect these in `explodeYieldExpression` where `getLineNumber` is already called.

### In transform output

Add to `TransformResult`:

```typescript
export interface TransformResult {
  code: string;
  errors: string[];
  sourceMap: PhaseMapEntry[]; // NEW
}
```

The `generateStepFunction` emits:

```javascript
export const __sourceMap = [[0,4,"const x = await sh(\"git diff\")","sh"],[1,4,null,"sh-resume"],...];
```

### At runtime

```typescript
// run.ts loads:
const mod = await import(scriptPath);
const step = mod.step;
const sourceMap = mod.__sourceMap ?? [];

// Passed to runner:
run({ scriptName, scriptPath, step, sourceMap });
```

When building the `sh-error` yield context:

```typescript
const entry = sourceMap.find(([phase]) => phase === currentState.phase);
const context = entry ? { line: entry[1], source: entry[2] } : { line: null, source: null };
```

## Resume flow changes

```typescript
// In resume handler (runner.ts or resume command):
function resume(opts: ResumeOptions): YieldMessage | DoneMessage {
  const meta = loadMeta(opts.sessionId);

  if (meta.pendingShError) {
    // This is a sh-error recovery resume
    if (opts.answer === "__abort__") {
      removeSession(opts.sessionId);
      throw new RuntimeError({
        error: { message: "Aborted by user", phase: -1 }
      });
    }

    // Execute new command
    const newResult = execSh(opts.answer, {
      timeout: meta.pendingShError.timeout
    });
    appendLog(opts.sessionId, {
      type: "sh",
      cmd: opts.answer,
      ...newResult
    });

    // Clear pending error flag
    clearPendingShError(opts.sessionId);

    // Feed to step at saved phase
    const state = loadState(opts.sessionId);
    return executeLoop(opts.sessionId, state, opts.step, JSON.stringify(newResult));
  }

  // Normal resume (existing logic)
  ...
}
```

## Edge cases

### Error in first step call (before any sh)

If `step()` throws on the initial call (phase 0, no input), this is NOT a sh error — it's a logic error in the script. Current behavior (RuntimeError) is correct.

### Error after sh result but NOT from sh-throws check

Example: script does `state.x = JSON.parse(input); state.y = state.x.stdout.trim()` and stdout is null → TypeError. This is NOT a sh error (no `.cmd` field on the thrown object). Current behavior applies.

### Multiple sh() calls before the failing one

The log already contains all previous sh executions. The `sh-error` yield includes the full log, giving the LLM complete context of what happened before the failure.
